# Implementation Plan: Deploy-Grade Dependency Resolution

**Branch**: `014-deploy-grade-dependency-resolution` | **Date**: 2026-03-25 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/014-deploy-grade-dependency-resolution/spec.md`

## Summary

Implement ADR-009's two-mode compilation architecture: keep the existing local `world` shim for fast authoring feedback, and add a deploy-grade compilation path that resolves the real upstream `world` package dependency graph via `@zktx.io/sui-move-builder`'s `resolveDependencies()` and `buildMovePackage()` APIs. This enables end-to-end compile → deploy → authorize turret flows against live testnet targets.

## Technical Context

**Language/Version**: TypeScript 5.9 (strict, ES Modules, no `any`)
**Primary Dependencies**: `@zktx.io/sui-move-builder` 0.2.8 (WASM), `@mysten/sui` 2.9.1, `@mysten/dapp-kit` 1.0.4, React 19
**Storage**: localStorage (deployment state persistence), in-memory (resolution cache)
**Testing**: Vitest + @testing-library/react (unit/integration), Playwright (E2E)
**Target Platform**: Browser (WASM compilation, wallet integration)
**Project Type**: Web application (single-page, client-side)
**Performance Goals**: Deploy-grade resolution ≤30s warm / ≤60s cold; authoring compile unchanged
**Constraints**: No server infrastructure; browser-only compilation; CORS-compatible fetching
**Scale/Scope**: 2 testnet targets, ~40 world source files per fetch, single user session

## Design Decisions

### D1. `MoveCompilerModule` Interface Widening

The current `MoveCompilerModule` interface in `moveCompiler.ts` is artificially narrow — it only accepts `files`, `silenceWarnings`, and `network`. The builder's actual `BuildInput` also supports `rootGit`, `resolvedDependencies`, `onProgress`, and more.

**Approach**: Widen the interface type to accept optional deploy-grade fields. The existing authoring path continues to pass only the fields it uses. The deploy-grade path passes the additional fields. This avoids a separate compiler wrapper.

### D2. Deploy-Grade Compilation as a Separate Module

Rather than modifying the existing `compileMove()` function, deploy-grade compilation will be a distinct function: `compileForDeployment()`. This keeps the authoring path untouched and makes the deploy-grade path independently testable.

**Justification**: Principle IX (Artifact Integrity & Lifecycle Separation) requires that authoring and deploy-grade compilation be treated as distinct concerns.

### D3. Extension Move.toml for Deploy-Grade

During authoring, the extension declares `world = { local = "deps/world" }`. For deploy-grade compilation, the extension's `Move.toml` must be rewritten to reference the world package by its git source identity, not a local path:

```toml
[dependencies]
world = { git = "https://github.com/evefrontier/world-contracts.git", rev = "v0.0.18", subdir = "contracts/world" }
```

The builder's `rootGit` input handles this mapping. The extension's `files` map includes the rewritten `Move.toml` plus the extension source (without shim sources). The builder fetches and resolves the world dependency via `rootGit`.

### D4. Fetching vs. Inline World Source

Even though `fetchPackageFromGitHub()` can fetch the world source, the deploy-grade path does NOT need to include world source files in the extension's `files` map. Instead, it provides `rootGit` so the builder resolves them automatically. The `fetchPackageFromGitHub()` function may be used to pre-fetch and cache the `Published.toml` for toolchain version checking and `original-id` extraction.

### D5. Published.toml Fetching for Metadata

For each target, the system needs `original-id` and `toolchain-version` from the upstream `Published.toml`. This file is on the `main` branch (not tagged) and is already fetched by `refreshPublishedWorldPackageManifest()`. That existing function will be extended to parse the additional fields.
