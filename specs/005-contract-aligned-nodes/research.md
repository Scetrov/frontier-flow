# Research: Contract-Aligned Nodes

**Feature**: 005-contract-aligned-nodes  
**Date**: 2026-03-17

## 1. Node Component Scaling Strategy

### Decision: Factory function pattern

The project currently has 9 node component files, each a thin wrapper around `BaseNode` that passes an icon. Scaling to 29 individual files is unnecessary because every node follows the exact same pattern: `(props) => <BaseNode {...props} icon={SomeIcon} />`.

**Chosen approach**: A `createNodeComponent` factory function in `src/nodes/createNode.tsx` paired with an icon-map lookup. The `flowNodeTypes` registry is generated by iterating `nodeDefinitions` and calling the factory for each entry.

**Rationale**:

- Single source of truth — adding a node requires only a definition entry and an icon-map entry
- Tree-shakeable — Vite eliminates unused icons at build time
- HMR-safe — changing the icon map only invalidates the factory module
- Testable — the factory is a pure function; output components render identically to handwritten ones
- Eliminates 20+ boilerplate files that would each contain exactly 6 lines

**Alternatives rejected**:

- **Individual files (status quo)**: Scales poorly; 29 files × 6 lines each, all identical except the icon import. No benefit to HMR granularity since changes to one node rarely affect another.
- **Single generic component with dynamic icon resolution**: Adds runtime overhead for icon lookup; harder to type-check; less explicit than a static map.

## 3. Vitest Testing Strategy for 29 Node Definitions

### Decision: `describe.each` parameterized pattern

The existing test file uses manual assertions for 9 nodes. At 29 nodes, this approach would be fragile and verbose. Instead, use Vitest's `describe.each` to iterate over `nodeDefinitions` and run structural, socket, and presentation tests for each.

**Pattern**:

```typescript
describe.each(nodeDefinitions)("$label (type: $type)", (definition) => {
  it("has required fields", () => {
    /* ... */
  });
  it("sockets are unique and typed", () => {
    /* ... */
  });
  it("colour uses CSS variable", () => {
    /* ... */
  });
});
```

Additionally, a cross-cutting snapshot test asserts the complete list of expected node types so that any addition or removal breaks the test cleanly.

**Rationale**:

- Each node gets its own describe block — test runner output groups failures by node
- Adding a new node definition automatically gains test coverage
- Snapshot of the canonical type list provides a "registry guard" against accidental changes

**Alternatives rejected**:

- **Manual per-node tests**: Does not scale; high risk of copy-paste drift
- **Loop with `it()`**: Works but produces flat test output; harder to diagnose which node failed

## 4. Existing Node Removal Strategy

### Decision: Replace in-place, update all dependents

The 9 existing node definitions will be replaced by 29 contract-aligned definitions. The 9 individual node component files (`AggressionNode.tsx`, `ProximityNode.tsx`, etc.) will be deleted in favour of the factory pattern.

**Impacted files**:

- `src/data/node-definitions.ts` — completely rewritten
- `src/nodes/index.ts` — rewritten to use factory
- `src/nodes/*.tsx` — all individual files deleted except `BaseNode.tsx`
- `src/nodes/createNode.tsx` — new factory file
- `src/__tests__/nodeDefinitions.test.ts` — rewritten with parameterized pattern
- `src/__tests__/canvasFlow.test.tsx` — update dropped node types to use new types
- `src/__tests__/Sidebar.test.tsx` — update fixture to use a new node definition
- `tests/e2e/sidebar.spec.ts` — no change needed (uses generic selectors)
- `src/components/CanvasWorkspace.tsx` — update empty-state copy

**Migration approach**: All changes land in a single atomic commit on the feature branch. No incremental migration needed since the old nodes have no persistent state.

## 5. Socket Type Compatibility

### Decision: No new socket types needed

All 29 node definitions use socket types already in the `SocketType` union: `target`, `tribe`, `boolean`, `number`, `list`, `priority`, `any`. No changes to the compatibility matrix are required.

**Verified**: Every socket type referenced in the spec is a member of:

```typescript
type SocketType =
  | "rider"
  | "tribe"
  | "standing"
  | "wallet"
  | "priority"
  | "target"
  | "boolean"
  | "list"
  | "number"
  | "string"
  | "any";
```

## 6. Category Ordering in Sidebar

### Decision: Sort definitions by category order in the data array

The spec requires sidebar ordering: event-trigger → data-accessor → logic-gate → data-source → action. Since the `Sidebar` component renders definitions in array order, the `nodeDefinitions` array will be sorted by category in this order. No grouping headers needed at this stage — the category label is visible on each node card.

**Rationale**: Avoids adding new sidebar UI components. The sidebar already renders all definitions in array order with category labels shown per card.
