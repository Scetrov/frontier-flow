# 1. Implementation Plan: Graph Import, Export, and Publish

**Branch**: `015-graph-io-publish` | **Date**: 2026-03-28 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/015-graph-io-publish/spec.md`

## 1.1. Summary

Add a versioned portable graph document pipeline around the existing saved-contract model, then surface three transfer flows in the current CanvasWorkspace drawer: import from YAML, export to YAML, and publish/import via Mysten Walrus. The implementation uses strict TypeScript adapters for graph document validation, the official Walrus SDK for browser-based blob storage, and a focused transfer UI that preserves the current Save, Save Copy, and Delete workflow.

## 1.2. Technical Context

**Language/Version**: TypeScript 5.9 (strict, ES Modules, no `any`)  
**Primary Dependencies**: React 19, `@xyflow/react` 12.10, `@mysten/sui` 2.9.1, `@mysten/dapp-kit` 1.0.4, `@mysten/walrus` 1.1.0, `yaml` for portable document encoding  
**Storage**: localStorage (contract library and lightweight provenance), downloaded YAML files, Walrus blob storage  
**Testing**: Vitest + Testing Library (unit/integration), Playwright (E2E for transfer UX)  
**Target Platform**: Browser SPA built with Vite, wallet-connected testnet usage for Walrus publish/import  
**Project Type**: Web application (single-project frontend)  
**Performance Goals**: YAML import/export under 5s for typical graphs; Walrus publish under 30s excluding wallet approval; no regression in existing drawer interactions  
**Constraints**: Browser-only implementation, Vite-hosted Walrus WASM, no backend publisher service, strict validation before mutating local graph state, wallet popup limitations may require explicit user-driven publish steps  
**Scale/Scope**: Single active graph at a time, local library of saved contracts, typical graph size up to 250 nodes / 400 edges, one Walrus blob per published graph

## 1.6. Design Decisions

### 1.6.1. D1. Portable Graph Document Mirrors a Single Saved Contract

The feature will not export the full contract library. Each import/export/publish action operates on the active saved contract only. The portable document contains a version marker, contract metadata, and the graph snapshot needed to reconstruct the saved contract without coupling the transfer format to unrelated local UI state.

### 1.6.2. D2. Validation Happens Before Any Library Mutation

All YAML and Walrus payloads flow through the same parser and validator before any local contract is created, renamed, or activated. This keeps imports deterministic, protects seeded examples, and centralizes all schema and graph consistency checks.

### 1.6.3. D3. Walrus Uses the Official SDK With Direct Blob APIs

The graph payload is a single YAML string, so the plan uses `readBlob` and `writeBlob` instead of Walrus quilt/file abstractions. The wrapper configures a Vite-resolved WASM URL and keeps Walrus-specific concerns out of `CanvasWorkspace`.

### 1.6.4. D4. Publish Flow is Explicit and User-Driven

Walrus browser flows can involve many requests and wallet-signing constraints. The publish UX therefore lives in a focused dialog with clear stages rather than a fire-and-forget button. If wallet prompts must occur in separated user interactions, the dialog can expose those steps cleanly without changing the underlying drawer layout.

### 1.6.5. D5. Published References Annotate, Not Replace, Local Contracts

After publish, the active saved contract stores optional Walrus provenance such as blob ID, publish timestamp, and network. The local contract remains editable and auto-saved as before. Walrus is treated as a transfer channel, not as the new source of truth.
